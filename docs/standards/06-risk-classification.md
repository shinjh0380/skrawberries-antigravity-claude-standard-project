# Standard 06: Risk Classification

## Purpose

태스크의 리스크 레벨을 일관되게 분류하여 적절한 계획 수준, 검증 강도, 에스컬레이션 절차를 결정합니다.

## Rules

1. **RISK-01**: 모든 구현 태스크는 시작 전 리스크 레벨을 분류해야 한다.
2. **RISK-02**: 리스크 분류는 `/scope-risk-pass` 스킬로 수행한다.
3. **RISK-03**: 리스크 레벨은 태스크 진행 중 상향 조정될 수 있으며, 하향 조정은 명시적 재평가가 필요하다.
4. **RISK-04**: `critical` 리스크는 사용자의 명시적 승인 후에만 진행한다.

## Risk Levels

### Low (낮음)
- 단일 파일, 자체 포함 변경
- 기존 패턴 반복 적용
- 데이터베이스/외부 API 영향 없음
- 예: 텍스트 수정, CSS 값 변경, 로직 없는 설정 변경

**요구사항**: 기본 검증, 빠른 구현 가능

---

### Medium (중간)
- 여러 파일에 걸친 변경
- 비즈니스 로직 수정 포함
- 내부 API 영향
- 예: 컴포넌트 props 추가, 유틸리티 함수 변경, 조건 로직 수정

**요구사항**: 계획 문서 작성, 관련 테스트 실행, 증거 번들

---

### High (높음)
- 아키텍처 수준 변경
- 데이터베이스 스키마 또는 마이그레이션
- 인증/보안 관련 코드
- 퍼블릭 API 변경
- 예: 새 테이블 추가, 인증 플로우 변경, API 엔드포인트 재설계

**요구사항**: 상세 계획, 단계별 검증, 롤백 계획, 전체 테스트 실행

---

### Critical (치명적)
- 프로덕션 데이터 마이그레이션
- 주요 의존성 교체
- 보안 취약점 패치 (즉시 배포 필요)
- 레거시 시스템 통합

**요구사항**: 사용자 명시적 승인, 단계적 배포 계획, 모니터링 계획

## Risk Escalation Triggers

다음 상황에서 리스크 레벨을 즉시 상향 조정:
- 예상보다 더 많은 파일이 영향받는 경우
- 의존성 체인이 발견된 경우
- 기존 테스트가 실패하기 시작한 경우
- 보안 관련 패턴이 발견된 경우

## Owner

CC (평가) / Shared (결정)

## Enforcement

- **Skill**: `/scope-risk-pass` — 분류 실행
- **Template**: `templates/risk-escalation-note.md` — 에스컬레이션 문서화

## Exceptions

핫픽스 (`priority: critical`): 리스크 분류 후 즉시 진행 가능, 단 사용자 알림 필수
