# Standard 07: Ambiguity Resolution

## Purpose

요구사항이 불명확하거나 충돌할 때의 결정 트리를 정의합니다. 추측 기반 구현을 방지합니다.

## Rules

1. **AMBIG-01**: 추측으로 구현을 진행하지 않는다. 불명확한 요구사항은 해소 후 진행.
2. **AMBIG-02**: 모호성 발견 시 즉시 사용자에게 질문한다. 침묵하며 추측 금지.
3. **AMBIG-03**: 질문은 구체적이고 선택지를 제시해야 한다. "이거 어떻게 할까요?" 형식 금지.
4. **AMBIG-04**: 해소 불가능한 모호성은 가정(Assumption)으로 문서화하고 사용자 확인을 받아야 한다.
5. **AMBIG-05**: 요구사항 충돌 시 더 보수적인(덜 변경하는) 해석을 기본으로 한다.

## Decision Tree

```
요구사항 불명확?
  │
  ├─ YES: 단순 명확화로 해결 가능?
  │         ├─ YES: 질문 생성 → `/intake-refine` → 답변 후 진행
  │         │
  │         └─ NO: 아키텍처적 결정 필요?
  │                 ├─ YES: `risk-level: high`로 에스컬레이션
  │                 │       `templates/risk-escalation-note.md` 작성
  │                 │       → 사용자 명시적 결정 요청
  │                 │
  │                 └─ NO: 가정을 문서화
  │                        → 가정 목록 사용자에게 확인 요청
  │                        → 확인 후 진행
  │
  └─ NO: 진행
```

## Question Quality Standards

### 나쁜 질문 예시
```
"버튼을 어떻게 처리할까요?"
"이게 맞나요?"
```

### 좋은 질문 예시
```
"로그인 버튼 클릭 시 두 가지 방식이 가능합니다:
A) 클라이언트 측 검증 후 API 호출 (빠른 피드백, 보안은 서버 의존)
B) 직접 API 호출 후 서버 응답으로 검증 (일관성, 약간의 지연)
현재 패턴(A)을 유지할까요, 아니면 B로 변경할까요?"
```

## Assumption Documentation Format

```markdown
## Assumptions

| # | 가정 내용 | 근거 | 확인 필요 |
|---|----------|------|----------|
| 1 | 페이지네이션 기본값은 20개 | 기존 API 패턴 | Yes |
| 2 | 에러 메시지는 한국어 | 프로젝트 언어 설정 | No |
```

## Owner

CC (감지) / Shared (해소)

## Enforcement

- **Skill**: `/intake-refine` — 모호성 질문 생성
- **Manual**: 구현 시작 전 사용자 확인

## Exceptions

- 단순 오타/명백한 실수: 수정 후 알림
- 이전에 동일한 결정을 한 기록이 있는 경우: 기록을 인용하고 동일하게 처리
